Components

What I'm building

Not a polished SaaS — a source-available project building local automation components piece by piece.
Each card uses ✅ / 🚧 / 🔬 to mark its current verification state.

🎯

✅ Skill packages

Install from registry / GitHub URL / local path. package.toml, source-bound config (secrets.json), SHA256 verification.

🧠

🚧 Reflection loop

Recent chats → repeated intent / topic candidates stored. First-turn suffix surface ✅ verified, "yes" reply dispatches ✅, generated code stability 🚧.

🔒

✅ Sandbox + Permission

All skills run in goja. SSRF guard, hop-by-hop header block, dangerous ops (Shell.exec / Git.push) require approval in supervised mode.

🔗

🔬 Skill chaining

Pipeline of skills. Staged execution validated for some packages. A general-purpose chain DSL is still undecided.

🌐

🚧 Web search

DuckDuckGo (no-config) + Google API + Tavily option. Web.search + Web.fetch primitives. Source quality / verification ongoing.

🏠

✅ Local LLM

OpenAI-compatible local endpoints (Ollama, LM Studio). Model performance varies — small models often have tool-use limits.

Workflow

The basic flow

From the CLI or local Web UI — all data lives in a local SQLite file.

📦
1

Install a skill

kittypaw skill install <name> — registry / GitHub URL / local path all supported.

⚙️
2

Configure

kittypaw skill config <name> <key> <value> — or kittypaw setup for LLM/channel in one shot. Required fields are in the package's package.toml.

⏱
3

Run / schedule

kittypaw skill run <name> for manual, or cron triggers driven by the server → output flows through the channel adapter.
